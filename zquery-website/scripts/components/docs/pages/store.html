Store — Global State

Lightweight reactive state management with named actions, computed getters, middleware, subscriptions, and action history. Inspired by Vuex/Redux but without the boilerplate.

Setup

const store = $.store({
  state: {
    count: 0,
    user: null,
    todos: [],
  },
  actions: {
    increment(state)     { state.count++; },
    setUser(state, user) { state.user = user; },
    addTodo(state, text) {
      const raw = state.todos.__raw || state.todos;
      state.todos = [...raw, { id: Date.now(), text, done: false }];
    },
  },
  getters: {
    doubleCount: (state) => state.count * 2,
    doneCount:   (state) => state.todos.filter(t => t.done).length,
  },
  debug: true,  // logs dispatched actions to console
});

Config Options

OptionTypeDefaultDescription
stateobject | () => object{}Initial state. Use a function for factory pattern (useful with named stores).
actions{ name: (state, ...args) => void }{}Named mutation functions. First argument is always the reactive state proxy.
getters{ name: (state) => any }{}Computed derived values. Re-evaluated on each access from the raw state.
debugbooleanfalseWhen true, logs every dispatched action to the console: [Store] action(args).

Dispatching Actions

All state mutations should go through actions. This keeps the mutation logic centralized and trackable.

store.dispatch('increment');
store.dispatch('setUser', { name: 'Tony', role: 'admin' });
store.dispatch('addTodo', 'Write documentation');

// dispatch() returns whatever the action function returns
const result = store.dispatch('someAction', payload);
Direct mutation works but is discouraged. You can write store.state.count++ directly and it will trigger subscribers. However, this bypasses middleware, debug logging, and action history. Prefer store.dispatch() for traceability.

Reading State & Getters

store.state.count              // reactive — reads trigger nothing special
store.getters.doubleCount      // computed from raw state on each access
store.getters.doneCount        // always up-to-date

// Deep-cloned snapshot (safe to serialize / send over network)
const snap = store.snapshot();
console.log(snap);  // { count: 0, user: null, todos: [] }

Subscriptions

Subscribe to state changes with key-specific or wildcard listeners. Both return an unsubscribe function.

PatternCallback SignatureWhen it fires
subscribe('key', fn)fn(newValue, oldValue, key)Only when the specified key changes
subscribe(fn)fn(key, newValue, oldValue)On any state change (wildcard)
// Key-specific subscription
const unsub = store.subscribe('count', (val, old, key) => {
  console.log(key + ': ' + old + ' → ' + val);
});

// Wildcard subscription (fires on every state change)
const unsubAll = store.subscribe((key, val, old) => {
  console.log(key + ' changed to ' + val);
});

// Always unsubscribe when done
unsub();
unsubAll();
Note the different callback signatures: Key-specific callbacks receive (value, old, key), while wildcard callbacks receive (key, value, old). The key is in a different position. Both return an unsubscribe function.

Middleware

Middleware runs before each dispatched action. You can use it for logging, validation, or blocking actions.

store.use((actionName, args, state) => {
  console.log('[middleware]', actionName, args);
  // return false to BLOCK the action from executing
  if (actionName === 'delete' && !confirm('Are you sure?')) {
    return false;
  }
});
ParameterTypeDescription
actionNamestringName of the dispatched action
argsArrayArguments passed to dispatch() (excluding the action name)
stateProxyThe reactive state proxy (readable, writable)
returnfalse | voidReturn false to cancel the action

Named Stores

Create multiple isolated stores by passing a name as the first argument. Retrieve them later with $.getStore().

const users = $.store('users', { state: { list: [] }, actions: {
  addUser(state, user) { state.list = [...(state.list.__raw || state.list), user]; }
}});

const app = $.store('app', { state: { theme: 'dark' }, actions: {
  toggleTheme(state) { state.theme = state.theme === 'dark' ? 'light' : 'dark'; }
}});

// Retrieve from anywhere
$.getStore('users');   // returns the users store
$.getStore('app');     // returns the app store
$.getStore();          // returns the 'default' store (if created without a name)

State Utilities

MethodSignatureDescription
replaceStatereplaceState(newState)Replace the entire state object (deletes all keys, then assigns new). Triggers subscribers.
resetreset(initialState)Same as replaceState() but also clears the action history.
snapshotsnapshot()Returns a deep clone of the current state via JSON.parse(JSON.stringify()).
history(getter)Returns a copy of the action log: [{ action, args, timestamp }].
// Replace state entirely
store.replaceState({ count: 0, user: null, todos: [] });

// Reset state and clear action history
store.reset({ count: 0 });

// Inspect action history
console.log(store.history);
// [{ action: 'increment', args: [], timestamp: 1719000000000 }, ...]

Using Store in Components

The recommended pattern: subscribe in mounted(), trigger re-render in the callback, and always clean up in destroyed().

$.component('my-widget', {
  mounted() {
    // Re-render when 'count' changes in the store
    this._unsub = store.subscribe('count', () => {
      this._scheduleUpdate();
    });
  },

  destroyed() {
    // ALWAYS unsubscribe — prevents memory leaks and ghost renders
    this._unsub?.();
  },

  render() {
    return `
      <p>Count: ${store.state.count}</p>
      <p>Double: ${store.getters.doubleCount}</p>
      <button @click="inc">+1</button>
    `;
  },

  inc() {
    store.dispatch('increment');
  }
});
Tip — Reactive arrays: Due to how JavaScript Proxy works, array mutations like push() are not detected reliably. Instead, always reassign with a new array: state.todos = [...state.todos.__raw, newItem]. Access .__raw to get the underlying plain array before spreading.
Memory leak prevention: If a component subscribes to the store in mounted(), it must unsubscribe in destroyed(). Otherwise, the subscription keeps a reference to the component and continues firing even after the component is removed from the DOM.
