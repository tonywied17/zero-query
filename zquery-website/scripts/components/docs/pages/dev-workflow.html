Development Workflow

zQuery is designed for a fast, streamlined development experience. During development you work with standard ES modules directly in the browser — write code, save, and see results instantly. The CLI includes a purpose-built dev server with live-reload so your changes appear in real time.

Installation

The dev server is included in the zero-query npm package. Install it as a dev dependency, or globally to use zquery anywhere without npx:

# Per-project (recommended)
npm install zero-query --save-dev

# Or install globally
npm install zero-query -g

This gives you access to the zquery CLI (via npx zquery or just zquery if installed globally), which powers both the dev server and the CLI Bundler.

Starting the Dev Server

From your project directory (the folder containing index.html), run:

# Start the dev server (default port 3100)
npx zquery dev

# Custom port
npx zquery dev --port 8080

The server will print its URL and begin watching for file changes. Open the URL in your browser and start editing — the page updates automatically when you save.

Instant feedback. The dev server serves your ES modules as-is. The browser’s native import support handles module resolution, so you see changes the moment you save.

Targeting a Sub-Directory

If your project lives inside a monorepo or nested folder, pass the path as an argument:

# Serve a specific project folder
npx zquery dev path/to/my-app

# Works with relative or absolute paths
npx zquery dev ../client-app
npx zquery dev C:/projects/my-app

The server auto-detects index.html in whichever directory you point it at. This is the recommended pattern for scaffolded projects — stay in the repo root (where node_modules lives) and point the dev server at the app folder:

# Scaffold, then serve from the repo root
npx zquery create my-app
npx zquery dev my-app
Why stay at the root? Keeping your working directory at the repo root means your IDE can resolve index.d.ts from the zero-query package for full IntelliSense and type support. If you cd into the app folder, the IDE may lose access to the type definitions.

Live Reload

The dev server watches your entire project tree for changes. When you save a file the browser refreshes automatically:

File TypeBehavior
.cssCSS hot-swap — stylesheets are reloaded in-place without a full page refresh, preserving component state
Everything elseFull reload — the browser performs a fresh page load to pick up the changes

The watcher monitors all file types in your project tree (excluding node_modules, .git, dist, and .cache). Changes are also logged to your terminal so you can see exactly which files triggered a reload.

Error Overlay

The dev server includes a full-screen error overlay inspired by tools like Vite and Angular that surfaces errors directly in the browser — no need to switch to the terminal or dig through the browser console. Errors are caught at two levels:

Syntax Errors (pre-validated on save)

Every time you save a .js file, the server validates it for syntax errors before broadcasting a reload. If a syntax error is found:

  • The page does not reload — your current state is preserved.
  • A dark overlay appears with the error message, file path, exact line:column, and a code frame showing the surrounding source with a caret pointing at the error.
  • The terminal also logs the error in red for quick reference.

When you fix the error and save again, the overlay clears automatically and the page reloads normally.

Runtime Errors

Uncaught exceptions and unhandled promise rejections are also captured and displayed in the same overlay with the error type, message, originating file:line:column, and a cleaned-up stack trace.

SSE Error Events

The error overlay communicates via two additional SSE event types on the existing /__zq_reload channel:

EventPayloadDescription
error:syntaxJSON { type, message, file, line, column, frame }Sent when a JS file fails syntax validation on save. Triggers the overlay.
error:clearemptySent when the previously broken file passes validation. Clears the overlay.
Dismiss. Press Esc or click the × button to close the overlay at any time. It will also clear automatically on the next successful reload.

How Live-Reload Works (SSE)

The dev server uses Server-Sent Events (SSE) — a lightweight, one-way push channel from the server to the browser. Here’s the flow:

  1. When the server responds with index.html, it injects a small <script> block before </body> that opens an EventSource connection back to the server at /__zq_reload.
  2. A file-system watcher (fs.watch, recursive) monitors your project for changes, ignoring node_modules, .git, dist, and other build artifacts.
  3. When a file changes, the server pushes a named event (reload or css) over the SSE connection.
  4. The injected client script receives the event and either reloads the page or hot-swaps stylesheets.

SSE Event Types

EventTriggerBrowser Action
reloadAny non-CSS file saved (.js, .html, etc.)Full page reload via location.reload()
css.css file savedStylesheet hot-swap — existing <link> tags are re-fetched in-place, preserving component state
error:syntaxJS file fails syntax validationError overlay displayed, reload suppressed
error:clearPreviously broken file passes validationOverlay dismissed, normal reload resumes
Source files are never modified. The SSE client is injected only in the HTTP response — your original HTML stays untouched on disk. If you view source in the browser you’ll see the injected script, but your file on disk is unchanged.
Why SSE over WebSocket? SSE is natively supported in all modern browsers, requires no handshake upgrade, reconnects automatically on disconnect, and needs zero client-side dependencies. For a one-way "server tells browser to reload" channel it’s the simplest and most reliable choice.

SPA Fallback Routing

The dev server includes built-in SPA support. Any request that doesn’t match an existing file is served index.html, so deep-link routes like /docs/router or /about work correctly during development without additional configuration.

How It Works

When a request comes in, the server checks for a matching file on disk. If no file exists (e.g. /about is a client-side route, not a real file), the server responds with index.html instead of a 404. The zQuery router then reads the URL and renders the correct component.

What’s Excluded

Static assets are always served directly — the fallback only applies to requests that don’t resolve to an existing file. Requests for .js, .css, images, and other static files are served normally.

Production Equivalents

In production you’ll need to configure the same fallback on your web server:

ServerConfiguration
ApacheFallbackResource /index.html in .htaccess
Nginxtry_files $uri $uri/ /index.html
Netlify/* /index.html 200 in _redirects
Vercel"rewrites": [{ "source": "/(.*)", "destination": "/index.html" }]
Note: SPA routing itself is a client-side feature of the zQuery router — it works everywhere, including production servers and file:// (via hash mode). The dev server simply provides the server-side fallback so you don’t need to configure Apache/Nginx rewrite rules during development. See the Router docs for more details.
Bundled apps on file://: When opening the dist/local/ build directly from disk, the router auto-switches to hash mode (#/about) so no server fallback is needed at all. See the CLI Bundler page for details.

Recommended Workflow

Summary: Use zquery dev as your primary development tool. It gives you the fastest feedback loop with instant reload. Run zquery bundle when you’re ready to deploy or distribute a production build.
