Development Workflow

zQuery is designed for a fast, streamlined development experience. During development you work with standard ES modules directly in the browser — write code, save, and see results instantly. The CLI includes a purpose-built dev server with live-reload so your changes appear in real time.

Installation

The dev server is included in the zero-query npm package. Install it as a dev dependency, or globally to use zquery anywhere without npx:

# Per-project (recommended)
npm install zero-query --save-dev

# Or install globally
npm install zero-query -g

This gives you access to the zquery CLI (via npx zquery or just zquery if installed globally), which powers both the dev server and the CLI Bundler.

Starting the Dev Server

From your project directory (the folder containing index.html), run:

# Start the dev server (default port 3100)
npx zquery dev

# Custom port
npx zquery dev --port 8080

The server will print its URL and begin watching for file changes. Open the URL in your browser and start editing — the page updates automatically when you save.

Instant feedback. The dev server serves your ES modules as-is. The browser’s native import support handles module resolution, so you see changes the moment you save.

Targeting a Sub-Directory

If your project lives inside a monorepo or nested folder, pass the path as an argument:

# Serve a specific project folder
npx zquery dev path/to/my-app

The server auto-detects index.html in whichever directory you point it at.

Live Reload

The dev server watches your entire project tree for changes. When you save a file the browser refreshes automatically:

File TypeBehavior
.cssCSS hot-swap — stylesheets are reloaded in-place without a full page refresh, preserving component state
Everything elseFull reload — the browser performs a fresh page load to pick up the changes

The watcher monitors all file types in your project tree (excluding node_modules, .git, dist, and .cache). Changes are also logged to your terminal so you can see exactly which files triggered a reload.

Error Overlay

The dev server includes a full-screen error overlay inspired by tools like Vite and Angular that surfaces errors directly in the browser — no need to switch to the terminal or dig through the browser console. Errors are caught at two levels:

Syntax Errors (pre-validated on save)

Every time you save a .js file, the server validates it for syntax errors before broadcasting a reload. The file is parsed line-by-line with import/export and import.meta expressions neutralized so the syntax check runs in a plain script context while preserving accurate line numbers.

If a syntax error is found:

  • The page does not reload — your current state is preserved.
  • A dark overlay appears immediately over the page with:
    • A color-coded badge (red for syntax errors, orange for runtime).
    • The error message, file path, and exact line:column number.
    • A code frame showing the surrounding source lines with a > marker on the error line and a ^ caret pointing at the exact column.
  • The terminal also logs the error in red with file and line info for quick reference.

When you fix the error and save again, the overlay clears automatically and the page reloads normally. The server tracks which file has an active error so the clear event is only sent when that specific file passes validation.

Accurate line numbers. The server strips ES module syntax (import/export) line-by-line and replaces import.meta expressions with harmless placeholders, so reported line and column numbers always match your original source file — not some transformed intermediate.

Runtime Errors

Uncaught runtime exceptions and unhandled promise rejections are also captured and displayed in the same overlay. Because the dev server serves your ES modules as-is (no bundling), file references in stack traces point directly to your source files:

  • The error type (e.g. TypeError, ReferenceError) and message.
  • The originating file, line, and column — pointing to your actual source, not a bundle.
  • A cleaned-up stack trace with the server origin stripped for readability. Internal overlay frames and SSE connection noise are filtered out automatically.

SSE Error Events

The error overlay communicates via two additional SSE event types on the existing /__zq_reload channel:

EventPayloadDescription
error:syntaxJSON { type, message, file, line, column, frame }Sent when a JS file fails syntax validation on save. Triggers the overlay.
error:clearemptySent when the previously broken file passes validation. Clears the overlay.
Dismiss. Press Esc or click the × button to close the overlay at any time. It will also clear automatically on the next successful reload.

How Live-Reload Works (SSE)

The dev server uses Server-Sent Events (SSE) — a lightweight, one-way push channel from the server to the browser. Here’s the flow:

  1. When the server responds with index.html, it injects a small <script> block before </body> that opens an EventSource connection back to the server.
  2. A file-system watcher (fs.watch) monitors your project for changes, ignoring node_modules, .git, dist, and other build artifacts.
  3. When a file changes, the server pushes a named event (reload or css) over the SSE connection.
  4. The injected client script receives the event and either reloads the page or hot-swaps stylesheets.
Source files are never modified. The SSE client is injected only in the HTTP response — your original HTML stays untouched on disk.

SPA Fallback Routing

The dev server includes built-in SPA support. Any request that doesn’t match an existing file is served index.html, so deep-link routes like /docs/router or /about work correctly during development without additional configuration.

Note: SPA routing itself is a client-side feature of the zQuery router — it works everywhere, including production servers and file:// (via hash mode). The dev server simply provides the server-side fallback so you don’t need to configure Apache/Nginx rewrite rules during development. See the Router docs and Production Deployment in the README for server config examples.
