Component System

Declarative components using template literals with reactive state, event delegation, two-way binding, scoped styles, and lifecycle hooks.

Defining a Component

$.component('app-counter', {
  // State: function returning object (recommended) or plain object
  state: () => ({ count: 0, step: 1 }),

  // Lifecycle hooks
  init()      { /* runs in constructor, before first render */ },
  mounted()   { /* after first DOM insert — safe to access refs */ },
  updated()   { /* after each subsequent re-render */ },
  destroyed() { /* cleanup: unsubscribe, remove timers, etc. */ },

  // Instance methods — available as this.method() and in @event bindings
  increment() { this.state.count += this.state.step; },
  decrement() { this.state.count -= this.state.step; },
  reset()     { this.state.count = 0; },

  // Template (required) — return HTML string
  render() {
    return `
      <h2>Count: ${this.state.count}</h2>
      <button @click="decrement">−</button>
      <button @click="reset">Reset</button>
      <button @click="increment">+</button>
      <input z-model="step" type="number" min="1">
    `;
  },

  // Scoped styles — auto-prefixed with a unique attribute selector
  styles: `
    h2 { color: #58a6ff; }
    button { margin: 4px; }
  `
});

stylesstringNoCSS string auto-scoped to the component via a unique [z-s{uid}] attribute selector. A <style> tag with data-zq-component attribute is injected into the document head.

Mounting & Lifecycle

// Mount a component into a DOM element
const inst = $.mount('#app', 'app-counter');

// Mount with initial props
$.mount('#app', 'app-counter', { initialCount: 10 });

// Auto-mount all registered custom tags found in the DOM
// Scans document.body (or a given root) for matching tag names
$.mountAll();
$.mountAll(someContainer);  // scan within specific root

// Get an existing instance from an element
const inst = $.getInstance('#app');

// Destroy a component (calls destroyed() hook, removes DOM, cleans up)
$.destroy('#app');

// Get all registered component definitions
const defs = $.components();  // { 'app-counter': {...}, 'home-page': {...} }
// ES module equivalent: import { getRegistry } from './index.js';
How auto-mount works: When $.mountAll() runs, it scans the DOM for any tags matching registered component names (e.g. <app-counter></app-counter>). HTML attributes on the tag are extracted as props — object/array values are auto-parsed from JSON.

Directives are special attributes in your render() template HTML that zQuery processes after each render. The table below shows the most common directives used in components. For the full directive reference — including z-if, z-for, z-show, z-bind, z-class, z-style, z-html, z-text, event modifiers (debounce, throttle, once, self), and more — see the Directives section.

Event Arguments

You can pass arguments to event handlers directly from the template. Supported argument types: strings, numbers, booleans, null, and state references (state.*).

<!-- String argument -->
<button @click="setFilter('active')">Active</button>

<!-- Number argument -->
<button @click="remove(${item.id})">Delete</button>

<!-- Boolean / null -->
<button @click="setVisible(true)">Show</button>
<button @click="reset(null)">Clear</button>

<!-- State reference (reads current value at call time) -->
<button @click="sync(state.count)">Sync Count</button>
Tip: The state.* prefix in event arguments reads the value from this.state at the time of the click, not at render time. This is useful for passing the current state value to a method without embedding it statically in the template.

Two-Way Binding — z-model

The z-model directive creates a reactive two-way sync between a form element and a state property. When the user types or selects, the state updates — and when the state changes, the element updates. Changes are fully reactive: other parts of the template that reference the same state value re-render automatically.

Supported Elements

Element / TypeDOM PropertyEventValue Conversion
Text input / textarea / defaultel.valueinputString as-is
type="number" / type="range"el.valueinputNumber(el.value)
type="checkbox"el.checkedchangeBoolean
type="radio"el.valuechangeString (checked state set by value match)
<select>el.valuechangeString
<select multiple>el.selectedOptionschangeArray of strings
[contenteditable]el.textContentinputString

Basic Usage

Bind any input to a state key. The template can display the same value elsewhere — it updates live as the user types:

$.component('profile-form', {
  state: () => ({
    name: '',
    bio: '',
    age: 25,
    agreed: false,
    color: 'blue',
  }),

  render() {
    return `
      <div>
        <!-- Text input — updates this.state.name on every keystroke -->
        <label>Name</label>
        <input z-model="name" placeholder="Your name">
        <p>Hello, ${this.state.name || 'stranger'}!</p>

        <!-- Textarea -->
        <label>Bio</label>
        <textarea z-model="bio" rows="3"></textarea>
        <p>${this.state.bio.length} characters</p>

        <!-- Number — auto-converts to Number -->
        <label>Age</label>
        <input z-model="age" type="number" min="0">
        <p>Age: ${this.state.age} (type: ${typeof this.state.age})</p>

        <!-- Checkbox — syncs as boolean -->
        <label>
          <input z-model="agreed" type="checkbox"> I agree
        </label>
        <p>Agreed: ${this.state.agreed}</p>

        <!-- Select -->
        <select z-model="color">
          <option value="red">Red</option>
          <option value="blue">Blue</option>
          <option value="green">Green</option>
        </select>
        <p>Color: ${this.state.color}</p>
      </div>
    `;
  }
});
How it works: z-model writes to the reactive state proxy on every input event, which triggers a re-render. The component automatically preserves focus and cursor position on any focused input, textarea, or select inside the component during re-render — not only z-model elements. The element is relocated after the DOM rebuild using the first available identifier: z-model attribute → z-ref attribute → tag/type/name/placeholder combination. This means typing feels seamless even in plain @input-bound search fields.

Radio Buttons

Multiple radio buttons can bind to the same state key. The one whose value matches the state becomes checked:

state: () => ({ size: 'medium' }),

render() {
  return `
    <label><input z-model="size" type="radio" value="small"> Small</label>
    <label><input z-model="size" type="radio" value="medium"> Medium</label>
    <label><input z-model="size" type="radio" value="large"> Large</label>
    <p>Selected: ${this.state.size}</p>
  `;
}

Select Multiple

For <select multiple>, state should be an array — it syncs the selected option values as an array of strings:

state: () => ({ toppings: ['cheese'] }),

render() {
  return `
    <select z-model="toppings" multiple>
      <option value="cheese">Cheese</option>
      <option value="peppers">Peppers</option>
      <option value="olives">Olives</option>
    </select>
    <p>Toppings: ${this.state.toppings.join(', ')}</p>
  `;
}

Nested State Keys

z-model supports dot-notation paths for binding to nested state properties:

state: () => ({
  user: { name: '', email: '' },
  settings: { theme: 'dark', fontSize: 14 },
}),

render() {
  return `
    <input z-model="user.name" placeholder="Name">
    <input z-model="user.email" type="email" placeholder="Email">
    <input z-model="settings.fontSize" type="number">
    <p>${this.state.user.name} (${this.state.user.email})</p>
    <p>Font size: ${this.state.settings.fontSize}px</p>
  `;
}

Modifiers

Add boolean attributes alongside z-model to modify binding behavior:

ModifierAttributeEffect
Lazyz-lazyListen on change instead of input — value updates on blur / commit, not every keystroke.
Trimz-trimAutomatically .trim() whitespace from string values before writing to state.
Numberz-numberForce Number() conversion regardless of input type (auto-detected for type="number" / type="range").
<!-- Only update state on blur (not every keystroke) -->
<input z-model="search" z-lazy placeholder="Search...">

<!-- Trim whitespace -->
<input z-model="username" z-trim placeholder="Username">

<!-- Force numeric conversion on a text input -->
<input z-model="quantity" z-number placeholder="Qty">

<!-- Combine modifiers -->
<input z-model="price" z-lazy z-number z-trim>
Under the hood: z-model writes to the reactive state proxy, triggering a re-render. Focus and cursor position on any focused input, textarea, or select inside the component are automatically saved before the re-render and restored afterward — so typing feels natural even though the entire DOM is being rebuilt. The element is relocated via z-model attr → z-ref attr → tag/type/name/placeholder fallback. Modifiers are parsed from boolean attributes (z-lazy, z-trim, z-number) on the same element.

Complete Two-Way Binding Example

This example demonstrates every z-model feature working together — text, number, checkbox, radio, select, select-multiple, nested keys, and modifiers — with live output:

$.component('binding-demo', {
  state: () => ({
    user: { name: '', email: '' },
    age: 0,
    subscribe: false,
    plan: 'free',
    tags: [],
    notes: '',
  }),

  render() {
    const s = this.state;
    return `
      <h3>Form</h3>
      <input z-model="user.name" z-trim placeholder="Name">
      <input z-model="user.email" z-trim placeholder="Email">
      <input z-model="age" type="number" min="0">
      <label><input z-model="subscribe" type="checkbox"> Subscribe</label>
      <label><input z-model="plan" type="radio" value="free"> Free</label>
      <label><input z-model="plan" type="radio" value="pro"> Pro</label>
      <select z-model="tags" multiple>
        <option>javascript</option>
        <option>html</option>
        <option>css</option>
      </select>
      <textarea z-model="notes" z-lazy rows="3"></textarea>

      <h3>Live Output</h3>
      <pre>${JSON.stringify({
        name: s.user.name,
        email: s.user.email,
        age: s.age,
        subscribe: s.subscribe,
        plan: s.plan,
        tags: s.tags,
        notes: s.notes,
      }, null, 2)}</pre>
    `;
  }
});

Instance API

After mounting, the returned component instance exposes:

Property / MethodTypeDescription
inst.stateProxyReactive state — mutating any property triggers a re-render
inst.propsObject (frozen)Read-only props from parent or mount call
inst.refsObjectMap of z-ref names to DOM elements (refreshed each render)
inst.templatesObjectKeyed map of loaded templates (when using multi-templateUrl or pages config). With pages, the active page is always present; other pages fill in via background prefetch.
inst.pagesArray<{id, label}>Normalized page metadata (when using pages config)
inst.activePagestringActive page id derived from route param (when using pages config)
inst.setState(partial)functionBatch-update multiple state keys: Object.assign(this.state, partial)
inst.emit(name, detail)functionDispatch a CustomEvent (bubbles, cancelable) from the host element
inst.destroy()functionFull teardown: removes listeners, styles, DOM, and calls destroyed() hook

const inst = $.mount('#app', 'my-widget');
inst.state.count = 5;              // triggers re-render
inst.setState({ count: 5, step: 2 }); // batch update
inst.emit('change', { value: 1 }); // dispatches CustomEvent
inst.refs.myInput.focus();         // access z-ref elements
inst.destroy();                    // teardown

Props

Props are read-only data passed into a component from the outside. They are accessible via this.props inside render and methods.

$.component('user-card', {
  render() {
    return `<div class="card">${this.props.name} — ${this.props.role}</div>`;
  }
});

// Via $.mount() — pass an object
$.mount('#el', 'user-card', { name: 'Tony', role: 'admin' });

// Via HTML tag + $.mountAll() — attributes become props
// <user-card name="Tony" role="admin"></user-card>
// JSON attributes are auto-parsed:
// <user-card config='{"debug":true}'></user-card>
// → this.props.config = { debug: true }
Tip: When mounted via the router, components receive special props: this.props.$params (route parameters), this.props.$query (query string), and this.props.$route (full route object). See the Router section for details.

Scoped Styles

Component styles are automatically scoped to prevent leaking into other parts of the page. Behind the scenes, zQuery adds a unique attribute selector (e.g. [z-s1]) to every rule in your styles string and sets the same attribute on the component's root element.

$.component('fancy-card', {
  render() {
    return `<div class="card"><h2>Title</h2><p>Content</p></div>`;
  },
  styles: `
    /* These only apply inside this component's root element */
    .card { background: #1a1a2e; border-radius: 12px; }
    h2 { color: gold; }
    p { color: #aaa; }
  `
});
Under the hood: A <style data-zq-component> tag is injected into <head>. Each selector in your styles string gets prefixed with [z-s{uid}], e.g. [z-s1] .card { ... }. This means styles are truly scoped — the same class name in different components won't collide.

External Templates & Styles (templateUrl / styleUrl)

For larger components, you can load HTML templates and CSS from external files instead of defining them inline. This keeps your JS clean, enables editor syntax highlighting for HTML/CSS, and lets you organize components into dedicated folders.

Relative Path Resolution

Relative templateUrl, styleUrl, and pages.dir paths are automatically resolved relative to the component file. No extra configuration is needed — just use simple filenames or relative paths:

// File: scripts/components/widget/widget.js
$.component('my-widget', {
  templateUrl: 'template.html',          // → scripts/components/widget/template.html
  styleUrl:    'styles.css',             // → scripts/components/widget/styles.css
  // ...
});
Tip: zQuery auto-detects the calling module's URL at registration time, so relative paths just work. Absolute paths (/some/path) and full URLs are never affected. If you need to override the resolved base, pass a base string (e.g. base: 'scripts/shared/').

styleUrl — Load external CSS

Use styleUrl to load styles from a separate .css file. The CSS is fetched on first mount, cached, and scoped automatically — exactly like inline styles.

$.component('my-widget', {
  state: { title: 'Hello' },
  render() {
    return `<div class="widget"><h2>${this.state.title}</h2></div>`;
  },
  styleUrl: 'styles.css'                 // relative to this file
});
Tip: If both styles (inline) and styleUrl are defined on the same component, they are merged — inline styles are applied first, then external styles.

templateUrl — Load external HTML template

Use templateUrl to load the component's HTML from a .html file. The template uses {{expression}} interpolation to access component state:

$.component('my-widget', {
  state: { title: 'Hello', items: ['A', 'B'] },
  templateUrl: 'template.html',
  styleUrl:    'styles.css'
});

Template file (template.html):

<div class="widget">
  <h2>{{title}}</h2>
  <p>Item count: {{items.length}}</p>
</div>
How it works: Templates and styles are fetched once per component definition and cached globally. Multiple instances share the same cached content. If render() is also defined, it takes priority over templateUrl. Relative paths resolve relative to the component file automatically. Absolute paths and full URLs are used as-is.
Bundler note: When using the CLI bundler, external templates and styles referenced via templateUrl, styleUrl, and pages configs are automatically detected and inlined into the bundle. No changes to your component code are needed — the library checks the inline map before calling fetch().

Multiple Templates — templateUrl as object or array

templateUrl also accepts an object map or array of URLs. When multiple templates are loaded, they are available inside the component via this.templates — a keyed map you can reference in your render() function.

// Object form — keyed by name
$.component('docs-page', {
  templateUrl: {
    'router':     'pages/router.html',
    'store':      'pages/store.html',
    'components': 'pages/components.html',
  },
  render() {
    const page = this.props.page || 'router';
    return `<div>${this.templates[page]}</div>`;
  }
});

// Array form — keyed by index
$.component('multi-page', {
  templateUrl: ['pages/intro.html', 'pages/advanced.html'],
  render() {
    return `<div>${this.templates[0]}</div>`;
  }
});
Tip: All templates in the map are fetched in parallel on first mount and cached globally. Subsequent mounts of the same component are instant. This is ideal for documentation pages, multi-step wizards, or tabbed content where each tab loads from a separate HTML file.

Multiple Stylesheets — styleUrl as array

styleUrl can also accept an array of URLs. All stylesheets are fetched in parallel, concatenated, and scoped to the component — just like a single stylesheet.

$.component('my-widget', {
  styleUrl: [
    '../shared/base.css',
    'styles.css',
  ],
  render() { return '<div class="widget">Content</div>'; }
});

Additional Global Stylesheets — $.style()

For global/app-wide styles (resets, layout, themes), a <link rel="stylesheet"> tag in your index.html is recommended — it prevents FOUC most reliably. $.style() is for dynamically loading additional global (unscoped) stylesheet files at runtime — theme switching, override files, conditional styles, etc. Unlike component styleUrl (which scopes CSS to a single component), stylesheets loaded via $.style() apply to the entire page. Paths resolve relative to the calling file, just like component paths.

// Dynamically load a theme stylesheet
const dark = $.style('themes/dark.css');
// ... later
dark.remove();  // unloads the stylesheet

// Load multiple files at once
$.style(['reset.css', 'theme.css', 'layout.css']);

// Load an override file on top of global styles
$.style('overrides.css');
DetailDescription
Relative pathsResolved against the calling module's directory (auto-detected). Absolute paths and full URLs are used as-is.
IdempotentDuplicate URLs are ignored — calling $.style('app.css') twice only injects one <link>.
DOM elementEach stylesheet is injected as <link rel="stylesheet" data-zq-style> in <head>.
RemovalThe returned handle's .remove() detaches all injected <link> elements.
When to use: Use a <link rel="stylesheet"> in index.html for global styles (best FOUC prevention). Use $.style() to dynamically load additional CSS files (themes, overrides). Use styleUrl for styles scoped to a component. Use component styles (inline string) for scoped inline CSS.

Pages Config — Declarative Multi-Page Components

The pages option is a high-level shorthand for components that display multiple pages loaded from a directory (e.g. documentation, wizards, tabbed content). Pages are lazy-loaded — only the active page is fetched on first render, and remaining pages are prefetched in the background for instant navigation.

PropertyTypeDescription
dirstringDirectory path containing the page HTML files
paramstringName of the dynamic route parameter that determines the active page. Must match a :param segment in your route definition (see below).
defaultstringPage id to show when the route param is absent or doesn't match any item. Also used as the fallback when a URL points to an unknown page.
extstringFile extension (default '.html')
itemsArrayList of page ids (strings) or {id, label} objects
How param Connects to the Router

The param property tells the component which route parameter to read. When the router matches a URL like /docs/components against a route pattern like /docs/:section, it extracts { section: 'components' } and passes it as this.props.$params. The pages config reads $params[param] (i.e. $params.section) and sets this.activePage accordingly.

Use the fallback property on the route so the bare path and the parameterized path are handled by a single route definition. When the fallback path is matched, the param is undefined and the default page is shown:

// routes.js
$.router({
  routes: [
    { path: '/docs/:section', component: 'docs-page', fallback: '/docs' },
  ]
});
// /docs           → activePage = default ('getting-started')
// /docs/router    → activePage = 'router'

String items auto-derive labels from the id: 'getting-started' becomes 'Getting Started'. Use an object when you need a custom label:

// File: scripts/components/docs/docs.js
$.component('docs-page', {
  pages: {
    dir:     'pages',                    // → scripts/components/docs/pages/
    param:   'section',                  // ← reads from :section in the route
    default: 'getting-started',          // ← used when no :section param present
    items: [
      'getting-started',                 // label: 'Getting Started'
      'project-structure',               // label: 'Project Structure'
      { id: 'http', label: 'HTTP Client' },
      { id: 'utils', label: 'Utilities' },
    ],
  },

  styleUrl: 'docs.css',                  // → scripts/components/docs/docs.css

  render() {
    // this.pages      → [{id, label}, …]  (normalized metadata array)
    // this.activePage  → 'components' when URL is /docs/components
    // this.templates   → { 'getting-started': '…html…', … }

    return `
      <nav>
        ${this.pages.map(p => `
          <a class="${this.activePage === p.id ? 'active' : ''}"
             z-link="/docs/${p.id}">${p.label}</a>
        `).join('')}
      </nav>
      <main>${this.templates[this.activePage] || ''}</main>
    `;
  }
});
Flow summary: User clicks z-link="/docs/store" → router matches /docs/:section → extracts $params.section = 'store' → pages config reads param: 'section' → sets this.activePage = 'store' → render() displays this.templates['store'].
How it works: Under the hood, pages builds a URL map ({ id: 'dir/id.html', … }) and normalizes items into a {id, label} array. On first render, only the active page is fetched — the component renders as soon as that single file is ready. All remaining pages are then prefetched in the background so subsequent navigation is instant. The component exposes this.pages (metadata), this.activePage (current id from the route param), and this.templates (loaded HTML keyed by id).
Real-world example: This very docs page uses the pages config to load 11 API sections from individual HTML files under pages/, with route-driven section switching via /docs/:section.

Adding Interactivity to Pages

Page HTML files are static content by default — perfect for docs, help text, and reference material. But if a page needs interactivity, you can embed a component directly inside the HTML. The component system automatically initializes any custom elements it finds in rendered content:

<!-- pages/getting-started.html -->
<h2>Getting Started</h2>
<p>Follow the steps below to set up your project.</p>

<!-- Interactive component embedded in a static page -->
<install-wizard></install-wizard>

<h3>Next Steps</h3>
<p>Once installed, explore the <strong>Components</strong> section.</p>
// components/install-wizard.js
$.component('install-wizard', {
  state: () => ({ step: 1 }),
  render() {
    return `
      <div class="wizard">
        <p>Step ${this.state.step} of 3</p>
        <button @click="next">Next</button>
      </div>
    `;
  },
  next() { if (this.state.step < 3) this.state.step++; },
});
Tip: Keep page files as plain HTML for content. When you need interactive widgets, create a component and drop its tag into the page HTML. This keeps content files simple and reusable while still allowing rich interactivity where needed.
When to use pages vs. routes: Use pages for content that shares the same layout (sidebar nav, search, shared chrome) and only swaps the main content area. If a “page” needs its own layout, lifecycle, or completely different UI, make it a separate route + component instead — that’s what the router is for.
