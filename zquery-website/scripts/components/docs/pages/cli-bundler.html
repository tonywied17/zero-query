CLI Bundler

zQuery includes a zero-dependency CLI that compiles your entire app — ES modules, the library, external templates, and assets — into a single production-ready bundle. It outputs two builds in one step: a server/ build for deploying to any web server, and a local/ build that works straight from disk. No config, no flags — just point it at your app.

How It Works

The bundler auto-detects your entry point, embeds the zQuery library, resolves all ES module imports, inlines external templates, rewrites your index.html, and copies assets into a dist/ folder. No flags needed — just point it at your app.

Installation & Usage

# Per-project (recommended)
npm install zero-query --save-dev

# Or install globally to use zquery anywhere without npx
npm install zero-query -g

# Bundle your app (auto-detects entry from any .html with a module script)
npx zquery bundle

# Or point to an app directory from anywhere
npx zquery bundle my-app/

Output goes to dist/ next to your index.html, with two sub-folders:

  • ▶dist
    • ▶serverdeploy to your web server
      • index.htmlhas <base href="/"> for SPA deep routes
      • z-app.a1b2c3d4.jsreadable bundle (library + app + templates)
      • z-app.a1b2c3d4.min.jsminified bundle
      • styles/copied CSS assets
      • scripts/vendor/copied vendor assets
    • ▶localopen from disk (file://)
      • index.htmlrelative paths, no <base> tag
      • z-app.a1b2c3d4.jssame bundle
      • …same assets
Two builds, zero hacks: The server/ build includes <base href="/"> so deep-route refreshes (e.g. /docs/router) resolve assets from the root — required for server-deployed SPAs. The local/ build omits it so paths resolve relative to the HTML file — perfect for opening via file:// with zero console errors.

Bundling Your App

To get started, scaffold a new project with the CLI and then bundle it:

# Scaffold a new project
npx zquery create my-app

# Bundle for production
npx zquery bundle my-app/

# Open the local build directly from disk
start dist/local/index.html

# Or deploy dist/server/ to your web server

Optional Flags

FlagShortDescription
--out <path>-oCustom output directory (default: dist/ next to index.html)
--html <file>—Use a specific HTML file instead of the auto-detected one

Both flags are optional — the bundler works out of the box with zero configuration. They exist for edge cases like mono-repos or non-standard layouts.

Examples

# Output to a custom folder
npx zquery bundle my-app/ --out build/

# Use a specific HTML entry when you have multiple .html files
npx zquery bundle my-app/ --html admin.html

# Combine both
npx zquery bundle my-app/ -o release/ --html dashboard.html
Tip: The --out path is resolved relative to the app directory, not your current working directory. For example, --out build/ creates my-app/build/server/ and my-app/build/local/.

Under the Hood

  1. Entry detection — Strict precedence strategy:
    1. HTML files — index.html is evaluated first, then other .html files (root + one level deep).
    2. Module scripts within HTML — within each HTML file, a <script type="module"> whose src resolves to app.js is preferred; otherwise the first module script tag is used.
    3. JS file scan — if no HTML match, .js files (up to 2 levels deep) are scanned in two passes: first for $.router( (the canonical app entry point), then for $.mount(, $.store(, or mountAll(.
    4. Convention fallbacks — scripts/app.js, src/app.js, js/app.js, app.js, main.js.
    Custom file and folder names are fully supported.
  2. Import graph — Recursively resolves all import statements and topologically sorts them (leaves first).
  3. Module stripping — Removes import/export keywords, keeps declarations. Output is plain browser JS wrapped in an IIFE.
  4. Library embedding — Finds zquery.min.js in your project or the npm package. Auto-builds from source if not found anywhere.
  5. Template inlining — Detects templateUrl, styleUrl, and pages configs and inlines the referenced HTML/CSS files so file:// works without CORS issues.
  6. HTML rewriting — Replaces <script type="module"> with the bundle, removes the standalone library tag, and produces two output directories: dist/server/ (with <base href="/"> for web servers) and dist/local/ (relative paths for file://). Assets are copied into both.
  7. Minification — Produces content-hashed filenames (z-app.<hash>.js / .min.js) for cache-busting. Previous builds are cleaned automatically.

Hash Routing on file://

When the bundled app is opened via file://, the router automatically switches to hash-based routing (#/about instead of /about). No configuration needed — z-link attributes and programmatic navigation work identically.

Why Hash Mode?

Browsers block the History API on file:// origins — calling pushState throws a security error. Hash-based routing sidesteps this entirely because the fragment (#/path) never leaves the browser: no server request is made, so there's nothing to block. The bundled router detects file:// at boot and switches automatically.

Server vs Local Comparison

The same app behaves differently depending on how it’s served:

ModeURLHow It Works
Server (dist/server/)https://mysite.com/docs/routerHistory API — clean paths, requires server-side fallback to index.html
Local (dist/local/)file:///C:/app/index.html#/docs/routerHash mode — everything after # is the route; no server needed

The switch is transparent to your code. Links created with z-link, calls to $.router().navigate(), route guards, and params all work the same way in both modes.

What Stays the Same

  • z-link attributes — the router rewrites them for the active mode
  • $.router().navigate('/path') — programmatic navigation works identically
  • Route guards (beforeEnter, beforeLeave) — fire in the same order
  • Route params (:id) and query strings — parsed and available on $.router().params
  • Wildcard / fallback routes — matched using the same logic
Tip: To test the local build quickly, run start dist/local/index.html (Windows) or open dist/local/index.html (macOS). Double-clicking the file in your file explorer also works.

Tips & Best Practices

  • Use relative imports — import './components/home.js' (not bare specifiers like import 'home'). The bundler resolves the import graph by following these paths, so bare specifiers will fail.
  • One component per file — the import walker resolves each file once; splitting code this way keeps the bundle order deterministic and makes it easy to add or remove features.
  • import.meta.url — automatically replaced at bundle time so that base, templateUrl, and styleUrl paths continue to resolve correctly in the IIFE output.
  • External templates — templateUrl, styleUrl, and pages configs are inlined automatically; no code changes needed. The bundler reads and embeds the HTML/CSS content directly into the bundle string.
  • Vendor scripts — third-party files referenced via <script> tags in your HTML (e.g. Prism, analytics) are left untouched and copied to both output directories as-is. They are not merged into the app bundle.
  • Re-bundle anytime — previous dist/ contents are cleaned automatically on each build, so you can re-run zquery bundle without manually deleting old files.
  • Cache busting — the output filename includes a content hash (z-app.a1b2c3d4.js). When your code changes, the hash changes, so CDN and browser caches are invalidated automatically.
  • No config file — there is no zquery.config.js or .bundlerrc. The CLI infers everything from your HTML and import graph. This keeps your project root clean and eliminates config drift.
Debugging bundle issues: If the bundler can’t find your entry point, run it with the --html flag pointing to your HTML file. Check that your <script type="module" src="..."> path is correct and uses a relative URL.
