Reactive Primitives

Low-level reactivity tools used internally by the component system and store. They are also exposed on $ for direct use in custom patterns — reactive proxies for objects, and a signals system for fine-grained reactivity.

Deep Reactive Proxy — $.reactive()

Wraps an object in a deeply reactive Proxy. Any property mutation (including nested) triggers the onChange callback.

const data = $.reactive(
  { user: { name: 'Tony' }, list: [1, 2, 3] },
  (key, value, old) => {
    console.log(key + ': ' + old + ' → ' + value);
  }
);

data.user.name = 'Updated';  // fires: "name: Tony → Updated"
data.list = [1, 2, 3, 4];   // fires: "list: ..." (reassignment)
ParameterTypeDescription
targetobjectThe object to wrap. Non-objects are returned as-is.
onChange(key, newValue, oldValue) => voidCalled on every set or delete operation, including deeply nested changes.

Special Properties on Reactive Proxies

PropertyTypeDescription
proxy.__isReactivebooleanAlways true on reactive proxies. Useful for checking if an object is already reactive.
proxy.__rawobjectAccess the underlying plain (non-proxy) object. Essential when you need to spread arrays: [...state.arr.__raw, newItem]
Array mutation caveat: Methods like .push(), .splice(), and .pop() on reactive arrays may not trigger the onChange callback reliably. Always reassign the entire array to guarantee a notification: data.list = [...data.list.__raw, newItem]

Signals — $.signal()

A signal is a reactive value container. Reading .value inside an effect() or computed() automatically registers a dependency.

const count = $.signal(0);

// Read and write
console.log(count.value);  // 0
count.value = 5;           // triggers all subscribers

// Peek without tracking (doesn't register as a dependency)
count.peek();              // 5

// Manual subscription (independent of effects)
const unsub = count.subscribe(() => console.log('count changed'));
count.value = 10;  // logs: "count changed"
unsub();           // stop listening
MemberTypeDescription
.value (get)anyRead current value. Auto-tracks dependency if inside effect().
.value (set)—Set new value. No-op if new value === old value. Notifies all subscribers.
.peek()() => anyRead value without tracking (won't create a dependency).
.subscribe(fn)(fn) => () => voidManual subscription. Returns unsubscribe function.
.toString()() => stringReturns String(value) for template literal use.

Computed Signals — $.computed()

A computed signal derives its value from other signals. It re-evaluates automatically when any dependency changes.

const price = $.signal(100);
const tax = $.signal(0.1);
const total = $.computed(() => price.value * (1 + tax.value));

console.log(total.value);  // 110
price.value = 200;
console.log(total.value);  // 220  (auto-recomputed)
tax.value = 0.2;
console.log(total.value);  // 240
How it works: $.computed(fn) creates a signal, then wraps fn in an effect(). Every time a dependency signal's .value is read inside fn, the computed signal auto-subscribes. When any dependency changes, the effect re-runs and updates the computed signal's value.

Effects — $.effect()

An effect runs a function immediately and re-runs it whenever any signal read during execution changes. Useful for side effects like DOM updates, logging, or syncing external state.

const x = $.signal(1);
const y = $.signal(2);

// Runs immediately, then re-runs when x or y change
const dispose = $.effect(() => {
  console.log('sum:', x.value + y.value);
});
// Logs: "sum: 3"

x.value = 10;
// Logs: "sum: 12"

y.value = 20;
// Logs: "sum: 30"

// Stop tracking — effect will no longer re-run
dispose();
FunctionSignatureReturnsDescription
$.signalsignal(initial)SignalCreate a new reactive signal with the given initial value
$.computedcomputed(fn)SignalCreate a derived signal that auto-updates from dependencies
$.effecteffect(fn)() => voidRun a side-effect function, re-run on dependency changes. Returns a dispose function.
$.reactivereactive(obj, onChange)ProxyCreate a deep reactive proxy around an object
When to use signals vs. reactive proxy: Use signals for individual primitive values or when you want fine-grained subscriptions (e.g. a counter, a toggle, a computed total). Use reactive proxies for complex nested objects where you want a single onChange callback (this is what components and the store use internally).
