Recommended Project Structure

zQuery apps follow a simple convention: one HTML file, one boot script, one component per file. The default setup uses ES modules served directly by the browser — no build tools are needed beyond the initial node build.js to create the library bundle. An optional CLI bundler is available if you need a single-file build.

my-app/
  index.html                 ← entry point
  scripts/
    vendor/
      zQuery.min.js          ← built library
    app.js                   ← boot: imports, router, store
    app.css                  ← global styles (linked in index.html via <link rel>)
    routes.js                ← route definitions
    store.js                 ← global store config
    components/
      home.js                ← one file per component
      counter.js
      todos.js
      about.js
      not-found.js

Conventions

RuleReason
Place zQuery.min.js in vendor/Keeps third-party code separate from your app code
One component per file in components/Easy to find, easy to delete; filename matches component name (e.g. docs/docs.js not docs/index.js)
Component names must contain a hyphenRequired by the HTML custom elements spec (e.g. home-page, app-counter). Single-word names like counter will not work.
Store and routes at scripts/ rootShared across all components; imported from app.js
app.js is the single entry pointImports components, creates store, boots router — keeps orchestration in one place
Use <link rel="stylesheet"> in index.html for global stylesMost reliable FOUC prevention — browser loads CSS before first paint. Use $.style() to dynamically load additional CSS files (themes, overrides). Use component styleUrl for scoped styles.
Global $ — no per-file importsLoaded via the script tag before modules, available everywhere
Use relative imports (./)Required for the CLI bundler to resolve the import graph. Bare specifiers are not supported.

SPA Index HTML Template

<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <title>My App</title>
  <link rel="stylesheet" href="styles/styles.css">
  <script src="scripts/vendor/zQuery.min.js"></script>
  <script type="module" src="scripts/app.js"></script>
</head>
<body>
  <nav>
    <a z-link="/">Home</a>
    <a z-link="/about">About</a>
  </nav>
  <div id="app"></div>
</body>
</html>
Tip: Use z-link attributes instead of plain href on navigation links. The router intercepts these clicks to navigate without a full page reload, preserving SPA state.

Sub-Path Deployment & <base href>

When deploying under a sub-directory (e.g. https://example.com/my-app/), add a <base href> tag to your HTML <head>:

<head>
  <base href="/my-app/">
  <!-- scripts, styles, etc. -->
</head>

Auto-detection (default): The router reads <base href> automatically — no extra code needed. Just call $.router() as usual:

$.router({ el: '#app', routes, fallback: 'not-found' });

Explicit option: Or pass base directly to override auto-detection:

$.router({ el: '#app', base: '/my-app', routes });
Tip: Using <base href> is preferred because it also controls how the browser resolves relative URLs for scripts, stylesheets, images, and fetch requests — keeping all path configuration in one place.

Then configure server-side rewrites so all non-file requests serve index.html.

Server config required (ES module setup): SPAs need a catch-all rewrite rule. Without it, refreshing on /about returns a 404 from the server. For Apache use FallbackResource /index.html, for Nginx use try_files $uri /index.html. This is not needed when using the bundled dist/ output with hash routing.

Bundle Output Structure

The CLI bundler produces a dist/ folder with two self-contained sub-directories:

my-app/
  dist/
    server/                    ← deploy to your web server
      index.html               ← has <base href="/"> for SPA deep-route support
      z-app.<hash>.js          ← all app modules + library in one IIFE
      z-app.<hash>.min.js      ← minified version
      styles/                  ← copied CSS
      scripts/vendor/          ← copied vendor assets (if referenced in HTML)
    local/                     ← open from disk via file://
      index.html               ← relative paths, no <base> tag
      z-app.<hash>.js          ← same bundle
      ...                      ← same assets
  index.html                   ← original source (ES module setup)
  scripts/
    ...
server/ vs local/: The only difference is the HTML. server/index.html includes <base href="/"> so deep-route refreshes (e.g. /docs/router) resolve assets from the site root — required for server-deployed SPAs with history-mode routing. local/index.html omits the <base> tag so all paths resolve relative to the HTML file — correct for file:// with zero console errors. On file://, the router auto-switches to hash mode.
Development: Use npm run serve to start a local dev server with SPA routing, or npm run dev to watch and auto-rebuild the bundle as you edit.
