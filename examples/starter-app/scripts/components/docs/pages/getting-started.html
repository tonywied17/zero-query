Getting Started

zQuery is a lightweight, zero-dependency frontend library (~46 KB minified). It provides a jQuery-inspired selector API, reactive components, an SPA router, global state management, an HTTP client, and utility helpers — all from a single script tag. Works out of the box with standard ES modules — no npm install or transpiler required. An optional CLI bundler is available for single-file distribution.

Quick Start (ES Modules — Recommended)

The preferred way to use zQuery is with the pre-built browser bundle (zQuery.min.js) loaded via a <script> tag, with your app code served as standard ES modules. No npm install, no transpiler, no build step. Grab the latest build from GitHub and drop it into your project:

Download from GitHub Download dist/zquery.min.js from the repo, or clone and run node build.js

1. Get the Library

git clone https://github.com/tonywied17/zero-query.git
cd zero-query
node build.js
# Creates dist/zQuery.js     (~78 KB, readable)
# Creates dist/zQuery.min.js (~46 KB, production)

The build script concatenates all source modules into a self-contained IIFE that exposes $ and zQuery on the window object. Copy dist/zQuery.min.js into your project's scripts/vendor/ folder.

Note: node build.js and npx zquery build must be run from the zero-query project root (where src/ and index.js live). If you’ve added a build script to your own package.json, npm run build handles the working directory for you.

2. Include in HTML

<!-- Load zQuery globally ($ and zQuery available on window) -->
<script src="scripts/vendor/zQuery.min.js"></script>

<!-- Your app code as ES module -->
<script type="module" src="scripts/app.js"></script>
Load order matters: The zQuery.min.js script tag must appear before your module scripts so that $ is available globally when your components execute.

3. Boot Your App

// scripts/app.js
import './components/home.js';
import './components/about.js';
import { routes } from './routes.js';

// Global styles are loaded via <link rel="stylesheet"> in index.html (recommended).
// $.style() is available for dynamically loading additional CSS files at runtime.

const router = $.router({
  el: '#app',
  routes,
  fallback: 'not-found',
});

$.ready(() => {
  console.log('zQuery v' + $.version + ' loaded');
});

The app.js file is the single entry point: it imports all component files, defines routes, creates the store (if needed), and boots the router. Component files register themselves via $.component(), so importing them is enough.

4. Define a Component

// scripts/components/home.js
$.component('home-page', {
  state: () => ({ greeting: 'Hello, World!' }),

  render() {
    return `
      <div class="card">
        <h2>${this.state.greeting}</h2>
        <button @click="update">Change Greeting</button>
      </div>
    `;
  },

  update() {
    this.state.greeting = 'Welcome to zQuery!';
  }
});
Tip: Component names must contain a hyphen (e.g. home-page, app-counter). This follows the HTML custom element convention and prevents collisions with native tags.

ES Module Imports (Advanced)

If you prefer ES modules over the global $, the source entry (index.js) exports everything as named exports:

import { $, zQuery, ZQueryCollection, queryAll,
  reactive, signal, computed, effect,
  component, mount, mountAll, getInstance, destroy,
  createRouter, getRouter, createStore, getStore,
  http, debounce, throttle, pipe, once, sleep,
  escapeHtml, html, trust, uuid, deepClone, deepMerge,
  isEqual, param, parseQuery, storage, session, bus
} from './index.js';
Key point: Most apps should use the built bundle (zQuery.min.js) which exposes $ and zQuery globally. Your component files use $ directly — no imports needed for the library itself. The ES module exports are available for advanced setups or tree-shaking bundlers.

CLI Bundler (Optional)

zQuery includes a zero-dependency CLI that compiles your entire app — ES modules, the library, external templates, and assets — into a single bundled file you can open directly from disk. Useful for offline distribution, file:// deployments, or reducing HTTP requests.

How It Works

The bundler auto-detects your entry point, embeds the zQuery library, resolves all ES module imports, inlines external templates, rewrites your index.html, and copies assets into a dist/ folder. No flags needed — just point it at your app.

Installation & Usage

# Install the package
npm install zero-query --save-dev

# Bundle your app (auto-detects entry from index.html)
npx zquery bundle

# Or point to an entry from anywhere
npx zquery bundle path/to/scripts/app.js

Output goes to dist/ next to your index.html, with two sub-folders:

dist/
  server/                    ← deploy to your web server
    index.html               ← has <base href="/"> for SPA deep routes
    z-app.a1b2c3d4.js        ← readable bundle (library + app + templates)
    z-app.a1b2c3d4.min.js    ← minified bundle
    styles/                  ← copied CSS assets
    scripts/vendor/          ← copied vendor assets
  local/                     ← open from disk (file://)
    index.html               ← relative paths, no <base> tag
    z-app.a1b2c3d4.js        ← same bundle
    ...                      ← same assets
Two builds, zero hacks: The server/ build includes <base href="/"> so deep-route refreshes (e.g. /docs/router) resolve assets from the root — required for server-deployed SPAs. The local/ build omits it so paths resolve relative to the HTML file — perfect for opening via file:// with zero console errors.

Bundling the Starter App

If you cloned the zero-query repository, bundle this starter app from the repo root:

# npm script (defined in package.json)
npm run bundle:app

# or equivalently
npx zquery bundle examples/starter-app/scripts/app.js

# open the local build directly from disk
start examples/starter-app/dist/local/index.html

# or deploy dist/server/ to your web server

Optional Flags

FlagShortDescription
--out <path>-oCustom output directory (default: dist/ next to index.html)
--html <file>—Use a specific HTML file instead of the auto-detected one
--watch-wWatch source files and rebuild on changes

What Happens Under the Hood

  1. Entry detection — Reads index.html for <script type="module" src="...">, or falls back to scripts/app.js, app.js, etc.
  2. Import graph — Recursively resolves all import statements and topologically sorts them (leaves first).
  3. Module stripping — Removes import/export keywords, keeps declarations. Output is plain browser JS wrapped in an IIFE.
  4. Library embedding — Finds zquery.min.js in your project or the npm package. Auto-builds from source if not found anywhere.
  5. Template inlining — Detects templateUrl, styleUrl, and pages configs and inlines the referenced HTML/CSS files so file:// works without CORS issues.
  6. HTML rewriting — Replaces <script type="module"> with the bundle, removes the standalone library tag, and produces two output directories: dist/server/ (with <base href="/"> for web servers) and dist/local/ (relative paths for file://). Assets are copied into both.
  7. Minification — Produces content-hashed filenames (z-app.<hash>.js / .min.js) for cache-busting. Previous builds are cleaned automatically.

Hash Routing on file://

When the bundled app is opened via file://, the router automatically switches to hash-based routing (#/about instead of /about). No configuration needed — z-link attributes and programmatic navigation work identically.

Tips

  • Use relative imports — import './components/home.js' (not bare specifiers like import 'home')
  • One component per file — the import walker resolves each file once
  • import.meta.url — automatically replaced at bundle time
  • External templates — templateUrl, styleUrl, and pages configs are inlined automatically; no code changes needed

Development Workflow

During development, the recommended workflow uses ES modules directly — no bundling needed:

# Start a local dev server with live SPA routing (uses zero-http)
npm run serve
# → http://localhost:3000

# Or use watch mode to auto-rebuild the bundle on every save
npm run dev

npm run serve starts a lightweight static server with SPA fallback routing — edit your source files and refresh the browser. npm run dev watches your source files and automatically re-bundles on changes (useful if you’re testing the bundled output).

Tip: Use npm run serve for the fastest feedback loop during development. Only bundle (npm run bundle:app) when you’re ready to deploy or distribute.
